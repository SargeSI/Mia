# Ведение бортжурналов автомобилей

## Назначение

Вести хронологию работ по пяти машинам автопарка Сергея в LightRAG. Каждая запись — в файл `vehicle-log-<ключ>.md`. Периодически (раз в 1-2 месяца) собирать из накопленных записей черновик поста для Drive2.

## Машины и ключи

| Ключ | Файл | VIN | Модель |
|---|---|---|---|
| `e53` | vehicle-log-e53-001-purchase.md и далее | WBAFB51040LU50083 | BMW E53 X5 4.4i N62 |
| `e39` | vehicle-log-e39-001-...md | WBADN21010GR72000 | BMW E39 535i |
| `suburban` | vehicle-log-suburban-001-...md | 3GNGK16FXVG188708 | Chevrolet Suburban K1500 6.5TD |
| `audi200` | vehicle-log-audi200-001-...md | WAUZZZ44ZLN076487 | Audi 200 20v |
| `rs6` | vehicle-log-rs6-001-...md | WUAZZZ4B44N900926 | Audi RS6 C5 |

## Структура записи

### Обязательные поля

- **Дата** — когда сделано (если не названа — ставить сегодняшнюю)
- **Машина** — ключ (e39/e53/suburban/audi200/rs6)
- **Пробег** — км на момент работы
- **Тип работы** — один из: `то`, `ремонт`, `диагностика`, `исследование`, `тюнинг`, `покупка`, `кузов`, `продажа`
- **Что сделано** — 1-3 предложения, суть
- **Запчасти** — список с артикулами и ценами, если применимо

### Опциональные поля

- **Диагноз** — симптомы → поиск → причина (для типов «диагностика» и «ремонт»)
- **Источники/ссылки** — где искали инфу, номера TIS, форумы, артикулы доноров
- **Цена работы** — если делал сервис
- **Кто делал** — сам / сервис
- **Фото** — факт наличия: «есть фото до/после» (галочка, не сами файлы)
- **Заметки на будущее** — отложенные работы, подозрения
- **Идея для поста** — если тянет на статью (true/false + тема одной строкой)

## Типы работ

| Тип | Когда применять | Идея для поста |
|---|---|---|
| `то` | Плановое ТО | Редко, только если нестандартное |
| `ремонт` | Сломалось → заменили/починили | Иногда |
| `диагностика` | Искали причину, процесс ещё не завершён | **Всегда** |
| `исследование` | Подбор запчастей, изучение совместимости, доноров, каталогов | **Всегда** |
| `тюнинг` | Улучшения сверх стока | Часто |
| `покупка` | Приобретение машины или крупного агрегата | Да |
| `кузов` | Покраска, рихтовка, антикор | Иногда |
| `продажа` | Продажа машины или запчастей | Да |

## Особое внимание: диагностика и исследование

Это два самых ценных типа записей. Информации по ним мало, каждая деталь важна.

### При записи диагностики фиксировать всю цепочку:

```
Симптом → гипотеза 1 (почему нет) → гипотеза 2 → проверка → результат → причина найдена / не найдена
```

Если диагностика привела к ремонту — связывать записи. При формировании черновика для Drive2 это готовая структура: «расследование».

### При записи исследования (подбор запчастей) фиксировать:

- Что ищем и на какую машину
- Рассмотренные варианты (доноры, аналоги, OEM, aftermarket)
- Артикулы с номерами
- Источники информации (TIS, RealOEM, форумы, опыт других)
- Что подходит, что нет и **почему**
- Открытые вопросы — что ещё нужно проверить перед покупкой
- Итоговое решение или статус «в процессе»

Если исследование завершено и запчасти куплены — добавлять запись типа `ремонт`.

## Логика уточнений

Цель: получить обязательные поля. Не допрашивать. Максимум 2 вопроса за раз.

Приоритет вопросов:
1. **Какая машина?** — если не ясно из контекста
2. **Пробег?** — всегда, если не назван
3. **Артикулы/цены?** — если была замена деталей
4. **Кто делал?** — если сервис
5. **Фото есть?** — если зрелищная работа

**Не спрашивать:** дату (ставлю сегодняшнюю, если не указано иное).

**Не додумывать:** если не сказано — поле остаётся пустым.

## Процедура сохранения в LightRAG

**КРИТИЧЕСКИ ВАЖНО:** Каждая запись — ОТДЕЛЬНЫЙ файл с уникальным именем. Никогда не перезаписывать файл с тем же именем (LightRAG даст статус `duplicated` и НЕ обновит содержимое).

**Формат имени файла:** `vehicle-log-<ключ>-<NNN>-<тема>.md`, где NNN — трёхзначный порядковый номер.

Примеры:
- `vehicle-log-e53-001-purchase.md` — Покупка E53
- `vehicle-log-e53-002-exterior.md` — Первый внешний осмотр E53
- `vehicle-log-e53-003-lift.md` — Осмотр на подъёмнике E53

**Порядок загрузки:**
1. Собрать запись по структуре ниже.
2. Загрузить через `mcp_LightRAG_upload_text` с УНИКАЛЬНЫМ `filename`.
3. Проверить статус: должен быть `success`. `duplicated` = использовать другое имя.
4. Дождаться обработки (`rag_system_status` → `processing: 0`).
5. Проверить поиском: `rag_query` с ключевыми фактами из записи. Если не находит — повторить с другим именем.

**Формат содержимого файла-записи:**
```
БОРТЖУРНАЛ BMW E53 X5 4.4i N62 — ЗАПИСЬ N: НАЗВАНИЕ

VIN: WBAFB51040LU50083
Дата: месяц год
Пробег: X км
Тип работы: покупка / диагностика / ремонт / исследование / тюнинг / кузов

РАЗДЕЛ 1:
Факты. Факты. Факты. Законченными предложениями. Без обрывков.

РАЗДЕЛ 2:
Ещё факты. Конкретные цифры, артикулы, цены — если известны.

ФОТО: описание того, что должно быть на фото
```

**Правила для LLM-читаемости:**
- Разделы заглавными буквами: `ДВИГАТЕЛЬ:`, `КУЗОВ:`, `ТОРМОЗНАЯ СИСТЕМА:`
- VIN явно указан в каждой записи
- Все факты — законченными предложениями
- Никаких намёков и недосказанностей — только то, что подтверждено
- Файлы не дополняются, не перезаписываются. Новая запись = новый файл.

## Питфолы

### LightRAG: duplicated при повторной загрузке

Статус `duplicated` = содержимое НЕ обновилось в индексе. Всегда использовать уникальное имя файла при каждом обновлении бортжурнала. Проверять поиском после загрузки.

### Близняшка не видит бортжурнал → выдумывает факты

Если близняшка (Cursor-Мия) или другой LLM-агент делает `rag_query` и не находит записей, он может начать сочинять факты на основе общей информации о машине (VIN-расшифровки, опции). Чтобы этого избежать:
- Записи бортжурнала должны быть самодостаточными и явно привязаны к VIN
- После каждой загрузки проверять, что rag_query находит свежие записи
- В промпте для близняшки явно указывать: «если бортжурнал не найден — НЕ выдумывай, скажи что данных нет»

### Всегда проверять LightRAG перед тем как спрашивать пользователя

Перед запросом VIN, пробега, деталей о машине — сначала сделать `rag_query` и/или `session_search`. Значительная часть информации уже есть в LightRAG (VIN, опции, история работ, артикулы). Запрашивать у пользователя только то, чего действительно нет в памяти.

### Не додумывать технические факты

Если нет точного подтверждения — указать степень уверенности. OЕ-поставщики, кросс-совместимость запчастей, расшифровка опций — проверять через поиск, а не выдавать догадки как факты. Особенно критично: расшифровка SA-кодов BMW, совместимость деталей между разными платформами.

### ОБЯЗАТЕЛЬНОЕ СОХРАНЕНИЕ НОВЫХ ФАКТОВ В LIGHTRAG

После КАЖДОГО сообщения пользователя, содержащего техническую информацию, новые факты, результаты диагностики или исследований — НЕМЕДЛЕННО сохранить их в LightRAG. Не дожидаться конца разговора, не откладывать. Критерии сохранения:
- Новая техническая информация (диагноз, артикул, параметр, технология)
- Результат исследования или сравнения
- Решение о выборе запчасти/модели
- Новые требования к проекту

Это НЕ опционально. Это жёсткое правило. Не сохранил = потерял.

### Справочные материалы

См. `references/bmw-coding-diagnostics.md` — BMW PA Soft / Scanner, ADFG, ZKE, опции кодирования.

## Работа с близняшкой (Cursor-Мия) для Drive2

Близняшка — отдельный LLM-агент, который не видит переписку. Ей нужно давать самодостаточный промт.

**Проверенный шаблон промта (3 запроса, все факты покрыты):**
```
Напиши черновик поста для Drive2 — [тема].

Сделай ТРИ отдельных rag_query в LightRAG:
Запрос 1: «[ключевые слова из записи 001]»
Запрос 2: «[ключевые слова из записи 002]»
Запрос 3: «[ключевые слова из записи 003]»

КРИТИЧЕСКОЕ ПРАВИЛО: только факты из найденных файлов. Если факта нет — НЕ выдумывай. Лучше пробел, чем ложь.

СТРУКТУРА, СТИЛЬ, ФОТО, ХЭШТЕГИ — как обычно.
```

**Почему 3 запроса:** один rag_query возвращает не весь документ, а только семантически релевантные чанки. Три запроса с разными ключевыми словами покрывают разные части бортжурнала. Имена файлов (vehicle-log-e53-001-purchase.md) использовать в запросе НЕЛЬЗЯ — LightRAG не ищет по имени файла. Только ключевые слова из содержимого каждой записи.

**Если близняшка всё равно выдумывает:** её rag_query не нашёл записи. Проверить свой rag_query с теми же словами — если не находит, перезагрузить файлы с новыми именами и другими ключевыми словами.

## Подготовка черновика для Drive2

Раз в 1-2 месяца или при накоплении 3+ неосвещённых записей по одной машине:

1. Запросить все записи по машине через LightRAG
2. Выбрать самую интересную тему (приоритет: диагностика/исследование > ремонт > тюнинг > ТО)
3. Написать черновик: техническая глубина, инженерный подход, без воды
4. Структура черновика: проблема → диагностика → решение → запчасти → результат
5. Места для фото пометить `[ФОТО: описание]`
6. Отдать Сергею на правку и добавление фото
