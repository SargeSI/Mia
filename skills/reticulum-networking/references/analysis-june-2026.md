# Reticulum: анализ ограничений и применимости (июнь 2026)

## Выводы сессии Сергей + Мия

### Что выяснили
1. Публичный testnet (Dublin, BetweenTheBorders) мёртв
2. Community-ноды существуют и работают (N7EKB: 47 клиентов, 32 ГБ трафика)
3. Reticulum через интернет требует белый IP хотя бы у одной ноды
4. NAT traversal отсутствует — архитектурное решение, не баг
5. В Москве mesh-покрытие LoRa практически отсутствует

### Когда Reticulum оправдан
- Off-grid связь (LoRa, packet radio) — основное предназначение
- Гетерогенные сети (radio + IP + I2P в одном стеке)
- Сценарии без интернета или с враждебным интернетом

### Когда Reticulum НЕ оправдан
- Простой чат с другом через интернет → self-hosted Matrix проще
- Замена Telegram/WhatsApp → другой класс задач
- P2P через NAT без своей инфраструктуры → не работает

### Ключевое архитектурное отличие от Jabber/Matrix
Transport node в Reticulum — рядовой узел без привилегированной роли.
Она не имеет доступа к содержимому (end-to-end шифрование).
При появлении прямого линка или альтернативного транспорта — нода исключается из маршрута автоматически.
В Jabber/Matrix сервер — архитектурная необходимость протокола.

### Практические рекомендации
- Для чата с другом: Matrix-сервер на своём VPS
- Для БПЛА/mesh: Reticulum как стек, своя transport node + LoRa-интерфейсы
- Reticulum и Matrix не конкуренты — разные ниши
