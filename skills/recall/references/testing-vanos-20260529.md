# Тестирование recall: запрос «ванос е39» — 29 мая 2026

## Контекст

BMW E39 535i (M62B35). В Obsidian есть заметка `ванос-стук-n63.md` с полным планом ремонта VANOS (адаптеры, обратные клапаны 11111734667, маслонасос 11411438280, датчик давления 12612367061, масло 10W-60).

## Ожидание

Запрос «ванос е39» должен найти BMW-сессию `cafbb586...` (E39_diagnostics).

## Результат

| Сессия | sim | Порог 0.5 |
|---|---|---|
| sess-cooking (Кулинария) | 0.5468 | ✅ ложное срабатывание |
| Born_Mia | 0.5084 | ✅ |
| cafbb586... (BMW E39) | **0.4337** | ❌ |

## Диагностика

### Прямой поиск через pgvector
```python
embedder.embed('ванос е39')
store.search_nearest(q_emb, threshold=0.0, limit=5)
# → Кулинария 0.5468, Born_Mia 0.5084, BMW 0.4337
```

### Причина
Embedding BMW-сессии создан по первому сообщению:
> «Ты профессиональный автомеханик, мне нужна твоя помощь. Есть у меня вот такой автомобиль @Авто/BMW E39 535i/BMW E39 535i.md»

Ни слова про VANOS. Модель qwen3-embedding:0.6b не вытягивает связь «ванос» ↔ «автомеханик + BMW E39».

### Дополнительно
Обсуждение VANOS велось через Cursor (другая сессия), попало в Obsidian, но не в pgvector. LightRAG находит (источники: `ванос-стук-n63.md`, `BMW E39 535i.md`), pgvector — нет.

## Выводы

1. **Embedding по первому сообщению не отражает содержание сессии из 200+ сообщений**
2. Нужно инкрементальное обновление embedding (скользящее среднее) по мере разговора
3. Нужно обогащение embedding контентом из Obsidian/LightRAG
4. LightRAG — надёжный fallback, компенсирует слепоту pgvector
5. Порог 0.5 даёт ложные срабатывания на коротких запросах (Кулинария > BMW по «ванос е39»)

## Решение (handoff близняшке)

Фоновое обновление эмбеддингов через `update_embedding.py` после каждого ответа. Не блокировать ответ пользователю.
