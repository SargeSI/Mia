## Протокол анализа

### Шаг 0. Определить PIR и горизонты

**PIR (Priority Intelligence Requirement)** — ключевой вопрос. Формулировка должна требовать **прогноза**, а не описания.

Примеры:
- «Какова вероятность прямого военного конфликта А и Б в ближайшие 30 дней?»
- «Ограничит ли страна X поставки ресурса Y в ближайший квартал?»
- «Введёт ли ЦБ ставку выше N% на ближайшем заседании?»

**Горизонты:** минимум два из [7d, 30d, 90d, 1y].

### Шаг 1. Фаза 0 — Числовой базис

**Загрузить релевантные скиллы из analytics/** (как минимум `analytics/factor-models`, `analytics/portfolio-risk-metrics`, `analytics/stress-tests-scenarios`).

Собрать количественные данные (минимум 5 показателей):
- Цена/уровень ключевого актива
- Волатильность (историческая, implied)
- Фундаментальные метрики (P/E, долг/ВВП, CDS-спреды)
- VaR/CVaR по релевантным позициям (если применимо)
- Консенсус-прогнозы или модельные оценки
- Сценарные параметры: шоки факторов (нефть ±X%, ставка ±Y п.п., рынок ±Z%)

**Источники:** Yahoo Finance, FRED, MOEX, ЦБ РФ, BIS, EIA. Для каждого показателя — дата и источник.

### Шаг 2. Фаза 1 — Событийная лента

Собрать конкретные факты (минимум 10 событий). **Каждое событие:** дата, источник, тема риска, Admiralty-рейтинг. Группировать по кластерам (перепечатки не считать отдельно).

**Загрузить `intelligence/admiralty-code`** — полная методика оценки источников и информации.

Краткая шпаргалка по Admiralty Code (детали — в скилле):
- Надёжность источника: A (полностью) … F (неизвестна)
- Достоверность: 1 (подтверждено) … 6 (неизвестно)
- Вес = reliability_weight × credibility_weight

### Шаг 3. Фаза 2 — Гипотезы и ACH

**Загрузить `intelligence/analysis-of-competing-hypotheses`** — полный протокол ACH (8 шагов по Heuer).

**Сформулировать минимум 3 взаимоисключающие гипотезы** (H1, H2, H3).

⚠️ **Питфол: status-quo bias.** Гипотеза «ничего не изменится, останется в диапазоне» (H2) почти всегда выигрывает формальный ACH — потому что большинство evidence с ней «consistent». Это не баг метода, а особенность: гипотеза «статус-кво» слишком широкая. 

**Лечение:**
1. После формального расчёта net_score — проверить, не является ли H2 «too easy» гипотезой (C по всем evidence, 0 противоречий)
2. Применить **фальсификационный подход:** ранжировать гипотезы по количеству противоречащих evidence (I + II), а не по net_score
3. Если H2 доминирует неоправданно — сузить её (например H2a: «верхняя половина диапазона», H2b: «нижняя половина»)
4. Дать **качественную корректировку** после формального расчёта — явно пометить: «Формальный ACH: H2. Качественная оценка: ... Причина: status-quo bias скорректирован.»

**Построить ACH-матрицу** (гипотезы × доказательства). Для каждой пары: CC, C, N, I, II.

**Расчёт net_score** (веса из `code/ach_mapping.py`):
```
support_weight   = 1.0 для CC, 0.5 для C, иначе 0
contradict_weight = 1.0 для II, 0.5 для I, иначе 0
score_support     = Σ(admiralty_weight × support_weight)
score_contradict  = Σ(admiralty_weight × contradict_weight)
net_score         = score_support − score_contradict
```

**Загрузить `intelligence/sherman-kent-estimative-probability`** — шкала Kent.

Маппинг net_score → Kent (baseline):
- < −0.3 → Unlikely (~10%)
- −0.3 … 0.0 → Probably not (~25%)
- 0.0 … 0.2 → Chances about even (~50%)
- 0.2 … 0.5 → Probable (~70%)
- > 0.5 → Almost certain (~90%)

Все выводы помечать: `scoring_mode: baseline`.

### Шаг 4. I&W — Ведущие индикаторы

**Загрузить `intelligence/indicators-and-warnings`** — методика I&W.
**Загрузить `intelligence/leading-indicators-economic`** — экономические опережающие индикаторы.

Для каждого сценария определить индикаторы. Типы: событийные, риторические, структурные, рыночные, макро, частотные (OSINT).

**Leading score:** `Σ(weight × triggered_flag) / Σ(weight)`, где triggered_flag: 0 / 0.5 / 1.

### Шаг 5. KAC — Проверка допущений

**Загрузить `intelligence/key-assumptions-check`** — полный протокол KAC.

Выявить неявные допущения. Для каждого: Assumption, Confidence (High/Medium/Low), Basis, Falsifier, Impact if wrong, Monitor. Критические допущения с уверенностью ниже «Probable» — пометить как **risk assumption**.

### Шаг 6. Milestones

Определить наблюдаемые события, при которых вывод нужно пересмотреть.

### Шаг 7. Сохранить результат в LightRAG

**Загрузить `lightrag-upload`** — процедура сохранения знаний.

После завершения анализа:
1. Сформировать самодостаточный документ из выходного шаблона (PIR, дата, горизонт, гипотезы, score, KAC, milestones)
2. Загрузить через `mcp_LightRAG_upload_text` с уникальным именем файла: `predictive-<slug>-<YYYY-MM-DD>.md`
3. Проверить статус загрузки (не `duplicated`)

Это критически важно для будущего пересмотра: когда сработает milestone или KAC-фальсификатор, анализ можно будет найти в LightRAG и пересмотреть.

### Шаг 8. Процедура пересмотра (Review)

Когда Сергей запрашивает пересмотр анализа или срабатывает milestone:
1. Найти предыдущий анализ в LightRAG: `rag_query("predictive <тема> <дата>")`
2. Загрузить актуальные данные (цены, события) — повторить Фазу 0 и Фазу 1
3. Проверить KAC-допущения: не появились ли falsifier'ы?
4. Проверить milestones: какие сработали?
5. Обновить ACH-матрицу с новыми evidence
6. Пересчитать net_score
7. Выдать обновлённый вердикт с пометкой «Review: <дата> ← Original: <дата>»
8. Сохранить новую версию в LightRAG с суффиксом `-v2`

## От сценариев к позиции (Bridge to Position)

Когда Сергей запрашивает анализ существующей позиции через призму предиктивных сценариев:

1. **Зафиксировать текущую позицию:** инструмент, количество контрактов, лот, цена входа, текущая цена, P&L
2. **Взять сценарии из предиктивного анализа** (H1, H2, H3 с вероятностями и целевыми уровнями)
3. **Для каждого сценария посчитать P&L** по каждому инструменту и суммарно
4. **Сформулировать варианты действий:** держать, сократить на X%, докупить на Y%, закрыть + перезайти, хеджировать
5. **Для каждого варианта посчитать:** средний вход, точку безубыточности, P&L по сценариям, максимальную просадку
6. **Вычислить risk/reward:** отношение прибыли в базовом сценарии к максимальной просадке
7. **Ранжировать варианты** по критериям: (а) не сжечь депозит, (б) не стоять против рынка, (в) не сидеть с лосями на просадке

**Принципы риск-менеджмента (из практики):**
- Докупка — консервативная: +10–20% к текущей позиции, не +50%. Просадка не должна угрожать маржин-коллом
- Усреднение входа — основная цель докупки, а не максимизация экспозиции
- Приоритет: сохранение депозита > участие в рынке > минимизация просадки
- Если точка безубыточности недостижима в базовом сценарии — позиция структурно слабая, а не «надо переждать»

## Протокол бэктеста

**Загрузить `intelligence/stress-tests-and-scenarios`** для связи сценарного анализа с ACH/KAC.

1. Выбрать ≥3 исторических кейса с известным исходом и значимым движением рынка
2. Собрать evidence и индикаторы за 7/14/30 дней до события
3. Рассчитать leading_score и net_score ретроспективно
4. Оценить: precision (≥0.5), recall (≥0.7), lead time (≥7d)
5. Скорректировать веса и пороги Kent
6. Переключить scoring_mode: baseline → calibrated

## Правила и ограничения

- **Глубина анализа:** привлекать события за неделю, месяц, квартал или год — в зависимости от PIR и горизонта
- **Исторические/структурные evidence:** прецеденты поведения актора — полноправный вид доказательств
- **Независимость:** разные типы evidence (news + market + macro) — независимые evidence. Перепечатки — один кластер
- **Это методичка, не ТЗ:** документ описывает метод мышления аналитика, а не спецификацию системы
- **Признавать неопределённость:** если данных недостаточно — так и сказать. Не додумывать
- **Scoring mode:** всегда указывать baseline или calibrated

## Питфолы

### ACH «ловушка статус-кво»
Формальный ACH-подсчёт даёт неоправданное преимущество гипотезе «ничего не изменится» (H2 = статус-кво). Почти все evidence оказываются «consistent» с ней, и она математически доминирует, даже когда не должна.

**Лечение:**
1. Проверить гипотезу статус-кво на ширину: если диапазон слишком широк, сузить или разбить на подгипотезы
2. Применить **фальсификационный тест** — считать только противоречия (I+II), игнорируя support
3. Если H2 имеет 0 противоречий при 10+ evidence — провести качественную корректировку
4. В вердикте явно указать: «Формальный ACH: H2. Качественная оценка с учётом [факторов]: ...»

### Однородные evidence из одного кластера
Перепечатки одного agency wire (AP → BBC → Guardian) без новой информации — один кластер, не три evidence. Считать их отдельно = завысить score_support.

### Смешивание горизонтов
Evidence для 7d (ультиматум) и evidence для 1y (структурный дефицит) — разные масштабы. Группировать evidence по горизонтам с пометками. При разборе позиции — разделять краткосрочный шок (недели) и среднесрочное восстановление (месяцы). Пример: Exxon CEO описал трёхфазную динамику (шок → восстановление потоков → компенсационный спрос) — нельзя всё смешивать в одну ACH-матрицу.

### Бюджетное правило как контринтуитивный фактор
При падении нефти Минфин ПРОДАЁТ валюту из ФНБ → рубль УКРЕПЛЯЕТСЯ → рублёвые нефтегазовые доходы падают ещё сильнее. Это проциклический механизм, который нельзя игнорировать в анализе рубля. Проверять: Urals относительно цены отсечки бюджетного правила.
