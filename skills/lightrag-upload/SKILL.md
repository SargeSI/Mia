# LightRAG Upload — процедура сохранения знаний

**Когда применять:** каждый раз, когда Мия узнаёт новый факт, получает файл, завершает исследование или находит важную информацию от Сергея.

## Что такое «знание»

**Знание** — самодостаточный, переиспользуемый факт с источником и контекстом, сохраняющий ценность за пределами текущего диалога.

Четыре критерия:

| Критерий | Проверка |
|---|---|
| **Переиспользуемость** | Пригодится ли через неделю? Месяц? |
| **Самодостаточность** | Понятно ли без контекста этого разговора? |
| **Верифицируемость** | Это факт/решение/договорённость, а не гипотеза? |
| **Источник** | Ясно ли, *кто*: Сергей, Мия, близняшка? |

**Знания** — загружать. **Не-знания** — нет.

## Алгоритм

1. **Оцени важность.** Спроси себя: «Пригодится ли этот факт через неделю? Через месяц?» Если да — загружай.

2. **Выбери метод:**
   - Текст (результаты поиска, договорённости, решения, инструкции) → `mcp_LightRAG_upload_text`
   - Файл (документы, скриншоты с текстом, PDF, код) → `mcp_LightRAG_upload_file`

3. **Сформулируй контент:**
   - Для `upload_text`: короткий самодостаточный документ (кто, что, когда, контекст, решение). Без отсылок вроде «как обсуждалось выше».
   - Для `upload_file`: убедись, что файл читаемый (PDF, MD, TXT, PY, JSON).

4. **Загрузи немедленно.** Не откладывай, не жди cron-синхронизации.

5. **Проверь:** после загрузки дождись статуса (processed, не failed).

## Что загружать

- Решения и договорённости с Сергеем
- Технические находки и решения проблем
- Результаты исследований (web_search, rag_query)
- Конфигурации и настройки
- Важные факты о проектах

## Что НЕ загружать

- Тривиальные сообщения («Привет», «Ок»)
- Промежуточные шаги задачи (в процессе выполнения)
- То, что уже есть в SOUL.md, MEMORY.md или USER.md (cron синхронизирует это раз в час)
- **Инфраструктурные изменения без явной команды.** Мия — аналитик, не исполнитель. Не создавай конфиги, не правь systemd, не запускай сервисы без явной команды Сергея. Обсуждай, изучай, спорь. К действию — только по команде.

## Примеры

**Симптом:** исследование завершено, но загрузка в LightRAG не произошла — агент переключился на следующую тему пользователя.

**Причина:** диалог увлекает. После получения результата хочется сразу ответить пользователю, а потом уже «не до того».

**Правило:** завершил исследование (3+ поисковых вызова, найден ответ, дана рекомендация) → **стоп** → загрузи результаты в LightRAG → только потом продолжай диалог или переключайся на следующую тему. Единственное исключение: пользователь явно просит продолжать без паузы.

**Почему это важно:** пользователь ожидает, что результаты будут доступны через rag_query в следующих сессиях. Если пропущена загрузка, вся работа по исследованию теряется для будущего использования. Пользователю приходится напоминать — это раздражает и подрывает доверие.

## Cron-синхронизация личности

SOUL.md, MEMORY.md и USER.md синхронизируются в LightRAG раз в час по cron-задаче. **Алгоритм:** см. `references/cron-sync-workflow.md`.

**Важно:** `upload_file(file_path=...)` НЕ РАБОТАЕТ для абсолютных путей к локальным файлам — MCP-сервер LightRAG не видит файловую систему хоста. Единственный надёжный способ: `read_file` → `upload_text(text=<content>, filename="<unique_name>.md")`. Это проверено и подтверждено на практике.

## Критический питфол: duplicated ≠ обновлено

При загрузке через `upload_text` с тем же `filename`, что и существующий документ, LightRAG возвращает статус **`duplicated`**. Это означает, что **содержимое в индексе НЕ обновилось** — осталась старая версия. Последующие `rag_query` будут находить старые данные.

**Решение:**
1. Всегда использовать **уникальное имя файла** при обновлении контента (суффикс с датой: `-2025-06-15`)
2. `upload_file(file_path)` НЕ РАБОТАЕТ для локальных путей — MCP-сервер их не видит. Используй `read_file` + `upload_text`
3. После загрузки проверять статус: `success` = ок, `duplicated` = НЕ обновилось
4. Проверить через `rag_system_status`, что документ обработан (processing: 0)
5. Финальная проверка: сделать `rag_query` с ключевыми словами из загруженного текста — убедиться что он находится

## Архитектура памяти

- **LightRAG** — долговременная память Мии и близняшки (граф знаний + векторный поиск). Сергей НЕ пользуется LightRAG.
- **Obsidian** — база знаний Сергея (Second Brain). Мия может читать для контекста, но не индексирует и не пишет без явной просьбы.
- Это РАЗНЫЕ ВСЕЛЕННЫЕ. Не «LightRAG = индекс Obsidian».
- **Cron** — отдельный механизм синхронизации личности (SOUL.md, MEMORY.md, USER.md), раз в час. Задачу сохранения знаний cron НЕ выполняет.
