# BR-20260603-04: Gateway впрыскивает RECALL-контекст чужой сессии в промпт агента

## Метаданные

| Поле | Значение |
|------|----------|
| **ID** | BR-20260603-04 |
| **Дата** | 2026-06-03 12:09 |
| **Автор** | Сергей (обнаружение), Мия (оформление) |
| **Компонент** | Gateway / Recall-маршрутизатор |
| **Серьёзность** | High — утечка данных между сессиями |
| **Статус** | Fixed — `_recall_route()` теперь возвращает clarify-сигнал вместо впрыска чужого контекста |

## Описание

Recall-маршрутизатор на уровне gateway впрыскивает `[RECALL: clarification ...]` с контекстом чужой сессии прямо в промпт текущей. Агент видит сообщения из сессии недельной давности в текущем диалоге, что приводит к сбою контекста.

## Шаги воспроизведения

1. Пользователь начинает новую тему в Telegram
2. Recall-маршрутизатор находит «похожую» сессию с sim 0.5–0.7
3. Вместо того чтобы спросить пользователя через платформу, впрыскивает `[RECALL: clarification — possible topic match: session=..., sim=0.62]` в промпт агента
4. Агент получает контекст чужой сессии, не понимает его происхождения и начинает отвечать в рамках ложного контекста

## Ожидаемое поведение

При sim 0.5–0.7 recall-маршрутизатор должен обратиться к **пользователю** через платформу (Telegram), а не к агенту через промпт.

Чужой контекст **не должен** попадать в промпт текущей сессии ни при каком значении sim.

## Фактическое поведение

В начале сессии агент видит в своём промпте:

```
[RECALL: clarification — possible topic match: session=20260527_012044_81b255ae, 
topics=Question about finalizing Hermes, sim=0.62. User will confirm or deny.]
```

Агент принимает контекст сессии недельной давности (про архитектуру Hermes) как релевантный и начинает отвечать в его рамках — хотя пользователь ведёт совершенно другой разговор.

## Контекст

- **12:09** — Сергей замечает глюк и пишет: «Ты не переключаешься в другую сессию, но в текущую сессию как-то попадают сообщения из другой сессии»
- **12:10** — Мия подтверждает: нашла `[RECALL: clarification ...]` в промпте, sim=0.62
- Это **не первый** инцидент — recall-инъекции происходили и раньше (утро того же дня: DPI/NFQWS → LightRAG/память)

## Гипотезы

| # | Гипотеза | Результат проверки |
|---|----------|-------------------|
| 1 | Recall-маршрутизатор не различает «спросить пользователя» и «впрыснуть агенту» | ✅ Подтверждена — `[RECALL: clarification ...]` попадает в промпт агента вместо сообщения пользователю через платформу |
| 2 | Это ошибка агента — не проверил релевантность | ❌ Опровергнута — агент не должен видеть этот контекст вообще |

## Корневая причина

`gateway/run.py` → `_recall_route()` при sim 0.5–0.7 впрыскивает `[RECALL: clarification ...]` в промпт агента вместо отправки сообщения пользователю через платформу.

Спецификация recall (скилл `recall`) предписывает:
- sim > 0.7 → переключить сессию
- sim 0.5–0.7 → спросить **пользователя**: «Ты про [topic1], [topic2] или это новая тема?»
- sim < 0.5 → ничего не делать

**Ни в одном сценарии** чужой контекст не должен попадать в промпт текущей сессии.

## Решение

Исправлено 03.06.2026 коммитом BR-20260603-04.

Изменения в `gateway/run.py`:
- `_recall_route()`: при `decision == "clarify"` возвращает `("clarify", candidates)` вместо `_build_recall_context([context], ..., label="CLARIFY")`
- `_build_recall_context()`: убрана ветка `CLARIFY` — остались только `RECALL` и `memory`
- Вызывающий код (строка 17343): при clarify-сигнале возвращает список кандидатов пользователю через платформу вместо впрыска в промпт агента

Чужой контекст больше не попадает в промпт текущей сессии.
